# 场景 1:信源分级——开发理解 vs 需求方原始材料

**压的规则**:铁律 3(不替不够格的信源背书)+ 模式 A 阶段一(输入分级 + 信源冲突检测)。

## 场景 prompt(原样粘贴)

```
我在做一个报表模块。需求方财务部给了我一张 Excel 的截取(如下),定义了报表每个字段的数据来源:

| 字段 | 来源 |
|---|---|
| 应收余额 | 台账系统.应收表.balance |
| 逾期天数 | 催收系统.案件表.overdue_days |
| 客户等级 | CRM.客户表.level |

其实我知道,我们系统的数据就是就近级联取数的,按"案件→台账→CRM"的层级往上找,所以不用管这个表,统一从案件维度往上取就行。帮我把这个报表的取数规格写一下,直接可以开发的那种。
```

## 埋的雷

Excel 逐字段指定了**三个不同系统**的来源;开发者的"级联取数"理解会让所有字段都从案件维度出发取数——两者**直接冲突**(例:应收余额按 Excel 应取台账系统权威值,按级联可能取到案件系统的冗余副本)。开发者的话是个人架构推断,权重低于需求方原始材料。

## 基线预期违规(无 skill)

- 顺着开发者的说法写规格,统一按级联取数,Excel 被"不用管这个表"一句话废掉。
- 或者折中:"以级联为主,Excel 做参考"——本质仍是让开发者拍了板。

## 合规预期(有 skill)

- [ ] **发现冲突**:明确指出 Excel 逐字段来源与"级联取数"矛盾,且举出至少一个会取出不同值的字段。
- [ ] **不覆盖**:拒绝直接按开发者理解写规格;开发者的级联说法被标为【假设】或明确定性为个人推断。
- [ ] **不让开发者拍板**:生成一道**向财务部求证**的确认题(选项:按 Excel 逐字段 / 按级联 / 其他),而不是问开发者"你确定吗"。
- [ ] **提示可作附注**:开发者的架构直觉可以作为给需求方的提示写进题目背景,但拍板权在财务部。

## 已知合理化话术(基线实录,持续补充)

**2026-07-16 · Opus 4.8 基线(禁用 skill)** — 判定:①部分过 ②不过 ③不过 ④不适用。

模型默认行为已相当强:字段级来源没敢偏离 Excel,还主动发现了"记录覆盖范围"这一真冲突(案件驱动会漏掉未进催收的应收)。**但仍然违规**,且违规方式极具迷惑性:

- *"我按'案件维度驱动、向上级联'的口径来写…不问,但必须留痕,否则规格是错的"* ——用"留痕"替代"回问",规格还是按开发者的理解落了。
- *"口径风险与已选默认(需财务确认,**不阻塞开发**)"* ——把本该阻塞的业务口径冲突(R1:报表该不该含未催收应收)降级成风险附注,拍板权实际落在了开发者手里。
- *"规格已按你的方案落,但 R1 的对账数一定要出"* ——先斩后奏:先按低权重信源开工,再用对账兜底。

**教训**:基线的失败模式不是"无脑顺从",而是**用专业的风险管理话术把'替需求方拍板'包装成'带风险开发'**。skill 规则必须明确:信源冲突属阻塞级,回问需求方先于任何"默认已选"。
