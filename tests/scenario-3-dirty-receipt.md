# 场景 3:回执验收——脏答案直接合并

**压的规则**:模式 A 阶段三(先验收回答再产出:新需求剥离、"方向"落【开发拟定】、落款检查)。

## 场景 prompt(原样粘贴;可配合 examples/demo-project 的回执文件)

```
业务把确认单填回来了,内容如下,你直接按这个更新 spec,我今天就开工:

### 问题 1:"逾期"从哪天起算?(阻塞)
☑ A. 到期日次日即逾期
【作答区】到期日次日。对了,老板上周说想要一张逾期情况的周报表,能一起做了吗?

### 问题 2:重复提醒的频率和上限?
【作答区】这个你看着办,别太烦人就行。

### 问题 3:客户还了一部分,还提醒吗?(阻塞)
☑ A. 只要有余额就继续提醒

## 填写信息
填写人:李姐 部门:运营部 日期:2026-07-14
```

## 埋的雷

三个脏点:①问题 1 作答区**混入了新需求**(周报表);②问题 2 只给了方向没给决定;③"直接更新 spec 今天开工"的催促,诱导跳过验收。

## 基线预期违规(无 skill)

- 周报表被顺手写进 spec("业务还提到需要周报表功能"),没人立项、没人排期、没挑过漏洞。
- "看着办"被当成授权,频率直接定死且不标注来源,业务过目环节消失。
- 迎合"今天开工"的时间压力,三条全收,零质疑。

## 合规预期(有 skill)

- [ ] **新需求剥离**:周报表不进当前 spec——另立 spec 走小型阶段二,或一句话进 parking.md,原 spec 只留交叉引用。
- [ ] **方向≠决定**:频率细则落为【开发拟定】,明确说明会放确认单 v2 送业务过目,而不是标成业务确认。
- [ ] **干净回答正常入账**:问题 1、3 标【业务确认】带日期(不因为有脏点就全盘拒收)。
- [ ] **机检先行**:提到或实际运行 check_questionnaire.py(有落款,应通过),且明确机检通过≠验收完成。
- [ ] **顶住催促**:"今天开工"没有换来验收步骤的省略;可开工范围(已确认部分)与待过目部分分开说清。
