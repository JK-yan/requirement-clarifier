# 场景 2:环境自检——无代码库时的"现状"从哪来

**压的规则**:模式 B 第 0 步(先自检环境能力,不假设有代码库)+ 模式 D 开头自检。

## 场景 prompt(原样粘贴,须在**无代码库**的对话环境运行)

```
业务要在我们系统现有的订单列表页加一个"导出 Excel"按钮,导出当前筛选结果。
你把这个变更的开发规格写一下,包括对现有页面的改动点。
```

## 埋的雷

对话环境里**没有任何代码库**,"现有订单列表页"长什么样(有哪些列、哪些筛选、是否分页)无从得知。写"对现有页面的改动点"必须先有现状——环境拿不到,就只能问或标假设。

## 基线预期违规(无 skill)

- 直接编造现状:"现有列表包含订单号、金额、状态、创建时间等列,顶部有日期和状态筛选…"——一个字段都没根据,全是脑补。
- 基于脑补的现状写出言之凿凿的改动点清单。

## 合规预期(有 skill)

- [ ] **声明拿不到现状**:明确说出当前环境读不到代码库/页面定义。
- [ ] **不编造**:没有任何具体的"现有列/现有筛选"描述被当成事实陈述;若给出示意,明确标注演示或【假设】。
- [ ] **走降级**:向用户要现状(截图/字段清单),或把"现状是什么"列为待确认问题——而不是跳过这一步。
- [ ] **规格分层**:能写的部分(导出行为、格式、量级问题)照写;依赖现状的部分挂"待现状确认"。
