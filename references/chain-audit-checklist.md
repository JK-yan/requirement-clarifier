# 链路组合审计清单（模式 D 必读，代码库前提）

定位：盲区清单管"业务说的"，本清单管"代码做的"——把实现里**没人定义过的规则组合**暴露出来，变成业务问题或守卫修复项。不需要用户先报障。

## 第一步：三份盘点（审计的原料）

1. **实体状态盘点**：核心实体的显式状态字段 + **隐式状态**（被拆分 / 被标记 / 被红冲 / 有在途审批 / 部分回款 / 负额）。隐式状态最容易漏——它们没有状态字段，只能从关联数据推断，而守卫恰恰经常忘了它们。
2. **操作盘点**：所有会改写这些实体的入口——UI action、service 函数、审批通过/驳回钩子、定时任务、脚本/遗留直写路径。逐个标注守卫在哪一层（UI / 服务层 / DB 约束）。
3. **不变量盘点**：系统必须永远成立的守恒式，**钱的优先**（如"Σ叶子净额 = 原额 − 回款 − 标记 − 红冲""在途审批同单唯一"）。写不出不变量的模块，先去问业务——写不出本身就是待确认需求。

## 第二步：组合矩阵（用不变量当探针）

- **状态 × 操作**：每格三问——该拦吗？实际拦在哪层？不拦会破坏哪条不变量？
- **操作 × 操作（顺序对）**：A→B 和 B→A 都过。重点：A 产生的新数据，B 依赖的旧关联是否继承/重算。
- **空间收敛（防爆炸）**：优先审 ①钱的不变量涉及的组合 ②新功能 × 老功能交界 ③最近改动过的模块 ④审批/异步在途 × 一切同步操作。一次只审一族组合，全量分批。

## 第三步：七种缺陷模式（逐条对照，附识别问法）

1. **数据不继承**：派生实体（子单/副本）没带上父级关联，下游按"自己名下查"落空。问：每个"生成新记录"的操作，新记录下游要用的关联数据都复制或可回溯吗？
2. **守卫不对称**：同一风险的同族入口只挡了一个。问：每条守卫，把条件里的名词换成同族名词（红冲→标记）还成立吗？
3. **UI 有、服务层无**：前端禁了按钮，服务函数裸奔——双开页面/接口直调即绕过。问：每个 UI 禁用条件，服务层有对应检查吗？
4. **幂等短路静默失败**：防重逻辑命中了不该命中的旧数据，操作"成功"但什么都没发生。问：每个"已存在则跳过"，排除已作废/已红冲/已废弃的记录了吗？命中时报错还是静默？
5. **锁覆盖不全**：在途流程只锁了发起它的操作，没锁其他会写同一实体的操作——通过时前提已变，卡死或脏写。问：每把锁，列出所有写该实体的操作，逐个问"它查这把锁吗？"
6. **合法路径缺失**：某个真实业务场景在任何操作顺序下都无解。问：对每条不变量的合法终态，从常见初态是否至少存在一条可走的操作序列？
7. **遗留双体系**：同一概念两套实现，一套已无入口但字段/数据还活着——详情页永远为空或口径分裂。问：每个枚举/分类字段，当前哪条写入路径在喂它？

## 单侧仓库时（纯前端 / 纯后端）

跨层缺陷模式（UI有服务层无、锁覆盖、不变量核账）在单侧仓库只能看到一半。规矩：**看不到的一侧绝不推测其行为**（暴露未知 > 假装完整）。做法：
- 照常完成本侧盘点：前端仓库盘"操作入口 + UI 守卫清单 + 接口契约"；后端仓库盘"服务守卫 + 数据模型 + 不变量"。
- 产出**第三类分流：跨仓待核对清单**——格式同问题清单，收件人是另一侧仓库/团队（"前端在 X 处禁用了 Y，请核对服务层是否有对应校验，file:line"）。拿到另一侧仓库时，这份清单即那边审计的输入。
- context.md 只认一个正本仓库（建议放后端，不变量在那侧）；另一侧仓库只沉淀本侧系统事实，避免上下文分裂。

## 第四步：产出纪律（接回主流程）

- 每个发现标注：维度（逻辑/操作/入口/核账）+ 严重度 + 自证引用（`> 证据: 路径:行号 | "原文片段"`）。**交付前跑 `scripts/verify_evidence.py`，核验摘要附在产出末尾**——FAIL 的修正或降级【假设】。
- **分流**：纯技术加固（守卫补齐类）→ 直接给修复项和工作量；**规则未定义类**（"拆分后红票该冲谁"）→ 选择题化 + 成本标注，进业务问题清单，走模式 A 阶段三管道；**单侧看不全类** → 跨仓待核对清单。
- **沉淀**：审计结论进 `specs/<链路名>-audit.md`；盘出的系统事实（"发票只挂父单"）进 context.md 系统级约束——下次审计和下次新需求都直接复用，越审越快。
