# 冷启动导入:项目做到一半才开始用本 skill

真实用户很少从项目第一天开始沉淀——多数是被模糊需求坑过之后中途引入,手里已经攒了几个月的旧聊天记录、旧 PRD、会议纪要和口头共识。冷启动的目标:**把存量材料变成可信的 context.md 起点**,而不是把一堆来历不明的"大家都知道"直接灌进去。

## 核心原则:存量共识没有落款,一律先按【假设】入账

口头共识最大的问题是 provenance:谁说的、哪天说的、当事人现在还认不认,全都查无实据。**导入 ≠ 转正**——导入时标【假设】,通过"补签核对表"批量转正。

## 五步导入

1. **盘点**:让用户列出手头的存量材料(聊天导出、旧 PRD、会议纪要、邮件、老 Excel),按"哪个功能还活跃"排优先级——**不求一次导干净,先导当前正在开发/维护的功能相关的**,其余登记在 context.md 待办里。
2. **逐份归档**:原样存入 `raw/<原始日期>-<来源>.md`(逐字,规则同阶段一;文件名用材料的原始日期,不是导入日期)。
3. **三类提取**,分流处理:
   - **黑话/术语** → context.md 黑话表(黑话无需落款,当场生效)。
   - **疑似已拍板的决定** → 汇成一张**补签核对表**(格式同确认单第一部分:当时的问题/结论/出处 raw 行号),整表送业务核对落款。签回前一律【假设】;签回后逐条转【业务确认】(日期用签回日,备注原始日期)。
   - **现行系统行为** → 有代码库:走模式 D 链路审计,以代码为准;无代码库:按用户口述入账,标【假设】"待代码/业务验证"。
4. **冲突留案**:存量材料内部互相矛盾的(旧 PRD 说 A、后来群里说 B),不猜哪个是最终版——按模式 B 变更源判定处理:同源按时序后者胜,不同源生成二选一确认题挂进补签核对表。
5. **收口**:context.md 写入导入摘要(哪些材料已导、哪些没导、补签核对表发出日期);补签核对表走阶段三的回执流程(机检 + 验收)。

## 反模式

- ❌ 把"大家都知道的规矩"直接写成【业务确认】——没有落款的确认是伪确认。
- ❌ 一次性导入全部历史,context.md 膨胀成没人读的垃圾场——按活跃功能增量导。
- ❌ 替业务回忆("当时应该是定了 A")——记忆不是证据,进补签核对表让当事人自己认。
