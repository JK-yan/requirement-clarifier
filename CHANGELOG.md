# Changelog

## [2.5.0] - 2026-07-16

### Added

- **`tests/` 压测场景**：判断题规则的回归测试（superpowers 式 RED-GREEN）。3 个场景：信源分级（开发理解 vs 需求方 Excel）、环境自检（无代码库编造现状）、回执验收（脏答案直接合并）。每个含场景 prompt、基线预期违规、合规判定清单。场景 1 附真实基线实录（Opus 4.8 禁用 skill）——失败模式不是无脑顺从，而是用"风险留痕、默认已选不阻塞"话术把替需求方拍板包装成带风险开发。
- **`references/cold-start.md`**：冷启动导入。项目中途引入 skill 时的存量材料五步导入法；核心原则：存量口头共识无落款，一律先【假设】、经补签核对表批量转正。
- **验收用例桥**：`spec-template.md` 验收标准升级为业务可核对的白话用例表（当…做…应看到…，锚定决定编号，含反向用例，交付逐条打勾），闭环从开工前延伸到交付时。示例 spec 同步演示。

### Changed

- `SKILL.md`（v2.5）：
  - **信源冲突堵漏**（来自场景 1 基线压测的直接发现）：明确信源冲突属阻塞级，禁止以"按我的理解先做、风险留痕""默认已选、不阻塞开发"绕过回问需求方。
  - 持久文件区新增 context.md 归档规则（约 300 行阈值，旧决定移入 context-archive.md）。
  - 模式路由表新增冷启动导入入口；参考文件区补充 cold-start.md。

## [2.4.0] - 2026-07-16

### Added

- **`scripts/check_questionnaire.py`**：确认单回执机检。阶段三验收答案前先跑——机判未答题（阻塞级未答 FAIL）、多选提示、"我不清楚"未给知情人、第一部分未核对、落款缺失（填写人/日期缺失 FAIL）、模板内部注释残留。机检通过 ≠ 验收完成，成色分级与冲突检测仍由 AI 完成。
- **`examples/demo-project/`**：完整走查案例（逾期提醒）。从一句微信语音到可开工 spec 的全链路缩微版，含脏回答处理（新需求剥离进 parking、"看着办"落为【开发拟定】、代签风险挂待办），spec 通过 `verify_evidence.py` 核验（7 PASS / 0 FAIL），回执通过 `check_questionnaire.py` 机检。

### Changed

- `scripts/verify_evidence.py` 新增第 5 项机判：【业务确认】所在行缺日期 → WARN，`--strict` 下计为 FAIL（补齐铁律 4"业务确认必须带日期"的机检兜底）。
- `SKILL.md` 阶段三接入回执机检步骤；参考文件区补充新脚本与示例目录。

## [2.3.0] - 2026-07-16

### Changed

- 同步 `SKILL.md` 正文到 v2.3（2.1/2.2 为本地迭代版本，未单独发布）：
  - **信源分级**：铁律 3 扩展为"不替不够格的信源背书"——使用者本人的理解/推断/架构阐述一律按【假设】处理，权重低于需求方原始材料；模式 A 阶段一新增"输入分级 + 信源冲突检测"前置步骤；阶段三冲突检测扩展为三类信源比对。
  - **环境能力自检**：模式 B 新增第 0 步（不假设有代码库，按"有代码/用户口述/回问业务"三档降级取现状）；模式 D 开头新增代码库可读性自检，读不到则如实告知、不臆测代码行为。
  - **产出物随环境分形**：无代码仓库环境产出自包含的业务语言需求文档，有仓库环境才产出带 file:line 证据的实现层 spec。
  - **白话不失锚**：业务确认单要求每个问题锚定到具体对象（哪张报表/哪个字段/哪句原文），语气通俗但指代精确。
  - **元信息行也是规格**：逆向场景新增第 6 条，原始材料中的"数据范围/备注"类元信息行须与字段清单交叉比对，已回答的问题绝不再问业务。
- **定位放宽（放开入口、锁死动作）**：`description` 与正文第一段不再限定"业务方口述"单一入口——群聊碎片、只写正常流程的模糊 PRD、开发者自己的产品构想、藏在现有 Excel/老系统里的隐性规则均为合法输入；需求来自开发者自己时，开发者兼任需求方角色。README 同步此定位。
- 更新 `SKILL.md` frontmatter 版本号为 `2.3.0`。
