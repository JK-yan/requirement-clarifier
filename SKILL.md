# Requirement Clarifier（需求澄清器）v1.12

## 背景与定位

使用者是**开发者本人**，直接对接一个不懂 PRD、不懂技术的业务方。业务给出的需求是口语化、跳跃、充满隐含假设的。本 skill 的目标是**把开发和业务之间痛苦的来回压到最少**，并让每次对接的成果沉淀下来。

铁律：

- **业务方全程不碰 AI**，只回答开发者转达的问题。不要建议把 prompt 交给业务方。
- **暴露未知 > 假装完整。** 原文没说的，标"待确认"，绝不自己补一个看似合理的猜测——那会把坑藏进漂亮的结构里（垃圾进，精装垃圾出）。
- **回答要分成色，结论要贴溯源标签。** 进入规格的每条结论标三档之一：【业务确认】（明确决定，带日期）、【开发拟定】（业务只给方向如"按角色过滤"，细则由开发落成具体默认规则，须放进确认单送业务过目，无异议生效）、【假设】（未经核实的说法，如"理论上有专人维护""逻辑应该都一致"——不得作为开发依据）。这套分级对开发者自己的话同样适用。**标签必须原样使用【】全角括号格式——harness 的统计与机判按字面 grep，丢括号等于丢标签，时间再紧也不许简写。**
- **AI 不替人拍板。** 值不值得做、选哪个方案、优先级，是业务和开发的决策；skill 只提供信息、选项和代价。
- **语言跟随用户，业务术语保留原话。**

## 持久文件：docs/requirements/

```
docs/requirements/
├── context.md              # 业务上下文（黑话表、干系人、系统约束、全局已确认决定、待办风险）
├── specs/<功能名>.md        # 每功能一份落地规格，含确认溯源
├── rules/<来源名>.md        # 逆向规则文档（owner 验真勾选版）
├── raw/<日期>-<来源>.md     # 原始材料归档（PRD/聊天记录/会议纪要/回执/公式导出件）
└── changes.md              # 变更日志
```

不存在则首次使用时创建（创建前告知用户），模板见 `templates/`。**每次被触发，第一件事：读 context.md 和相关 spec**——已确认过的绝不重复问。**每轮结束时主动写回**：新黑话、新回答、新约束、新风险。沉淀不是可选项。

## 模式路由

| 用户输入特征 | 模式 |
|---|---|
| 贴了一段新需求 / "帮我理一下" | 模式 A：新需求澄清 |
| 需求载体是现成的 Excel/老系统/公式表 | 模式 A + 逆向场景方法（见下） |
| "业务说要改 XX" / 与已有 spec 冲突 | 模式 B：需求变更 |
| 带回了业务对问题清单的回答 | 模式 A 阶段三 |
| "把 XX 记一下" / 纠正术语 | 模式 C：上下文维护 |
| "排查/审计 XX 链路" / "还有没有类似的坑" / 无症状主动体检（需代码库） | 模式 D：链路审计 |

拿不准就问一句，不要猜。

## 模式 A：新需求澄清（三阶段）

### 阶段一：吸收 + 结构化

接住原始需求（口述、聊天记录、会议纪要、PRD，多乱都行）。**第一步先落盘**：原始材料存 `docs/requirements/raw/<日期>-<来源>.md`（**逐字粘贴，不许改写原话**；删减隐私只用 `[已删:原因]` 占位——raw 是全部证据的地基，改写过的原文等于伪造证据）。然后对照 context.md 翻译黑话，整理成开发视角结构：要解决的问题（业务原话）、输入、输出、处理逻辑、边界（做/不做）、异常。**结构化里每条"业务要求"必须自证引用归档件原文**（`> 证据: raw/xxx.md:行号 | "原话片段"`）——引用不出原文的条目就是模型的推断，一律标"待确认"或【假设】，绝不冒充业务说过。

### 阶段二：挑漏洞（核心步骤）

1. **读 `references/blindspot-checklist.md`**，逐维度过一遍结构化结果（权限、状态、并发、量级、存量、时序、删除语义等）。
2. **看代码库**（Claude Code 中）：现有数据模型、类似功能、约束。目的：(a) 代码能答的不烦业务；(b) 给选择题标注真实开发成本。**若需求是往老系统加新操作/新状态**，额外用 `references/chain-audit-checklist.md` 的组合矩阵审新老交界（新操作×老状态、老操作×新状态，七种缺陷模式当预防清单）——尤其别忘了反向：老守卫要不要挡新状态。审出的规则未定义项照常进问题清单。
3. 产出**问题清单**：

```
## 必须先确认（阻塞开发）
1. [问题] —— 涉及：[功能/字段]（盲区维度）（建议由 XX 回答）
   - 选项 A：...（成本/影响）
   - 选项 B：...（成本/影响）
   - 我的默认建议：...
## 隐含假设，需要业务确认
## 可以后补（不阻塞）
```

要求：只列业务才能回答的（技术选型开发自己定）；一律选择题化——业务擅长"选 A 还是 B"，不擅长从零描述；已有答案的不问；按阻塞程度排序；**每题标注建议回答人**——知情人常散在多拨人手里（提需求业务、财务、外围系统操作者、销售管理等），并警惕"替答"：别让提需求的人替财务或其他团队拍板。

**产出形式：可填写确认单。** 除对话内清单外，生成可直接发业务填写的文档（模板 `templates/questionnaire-template.md`，正式场合出 docx）：①开头附"已确认事项核对表"，把口头共识变成有落款的凭证；②每题 ☐ 选项 + 作答区；③"建议由 XX 回答，不归您管请转交"；④给"我不清楚"的台阶选项——比逼人瞎选强，且顺势索要真正知情人的名字。

### 阶段三：成型（先验收回答，再产出）

用户带回答案后，**不要直接合并**。先做两件事：

1. **对答案再跑一遍挑漏洞。** 回答常常不干净：混着**新需求**（"要一个展示计算过程的页面"）、答非所问（问"谁维护"答"根据权限"）、乐观假设（"逻辑应该都一致"）。新需求一律剥离——**剥离 = 另建独立 spec 文件走小型阶段二**（哪怕它自带冲突待决，冲突确认题也挂新 spec 下），原 spec 只留一行交叉引用。**剥离广度阀门**：同一轮验收剥离超过 2 个新需求、或下一轮回答仍在冒新需求 → 停止继续立项，其余新想法一句话一条归集进 `docs/requirements/parking.md`（需求停车场），请业务排完优先级再逐个立项——当前 spec 必须先收口；假设按【假设】入账。
2. **与已确认决定做冲突检测。** 每条新答案对照 context.md 和 spec 既有决定过一遍——新答案撞旧决定（如"计算过程页"撞"成本仅财务可见"）是高频事故，必须当场揪出、让业务二选一，绝不无声合并。

**收敛判据（何时停止追问）：** 当回答从"决定"退化为"方向"（按角色过滤/根据权限/看着办），说明问题已细到业务只能给方向——**停止追问**。例外：若"方向"指向别的知情人（"按财务平时的搞法"），这不是收敛信号而是路由信号，先转问该知情人（并把映射记进 context.md 干系人表）；路由穷尽仍只有方向时，才由开发把方向落成具体默认规则，标【开发拟定】，放进最终确认单送过目。继续追问只会得到更敷衍的回答。

**业务沉默协议：** 确认单发出时注明回填时限（默认 3 个工作日）。超时未回 → 提醒一次并抄送拍板人；再超时 → 【开发拟定】条目转"**超时临时生效**"：spec 与 changes.md 记录时间戳与"业务未回复，临时生效，风险归属需求方"，条目保持临时标记直至补落款——**超时生效不等于业务确认，标签永不因超时转正**；阻塞级问题不适用降级，只能推迟对应部分的开发或向拍板人升级。

**落款检查：** 回收确认单核对填写人/日期；回执（确认单/聊天记录/邮件）归档进 `raw/`，spec 确认记录的每条决定引用回执行号，使锚点可核验。缺落款提醒用户补。注意 provenance：聊天记录里业务"说过"≠"决定了"，随口畅想按【假设】入账。

验收后合并，产出**两份**：

**① 开发规格** → `docs/requirements/specs/<功能名>.md`（模板 `templates/spec-template.md`）。只写实现层：功能说明、输入/输出、处理逻辑、数据结构、边界、异常、验收标准。每条关键决定带溯源标签与日期。不写 ROI、灰度、里程碑——短闭环用不上。

**② 业务确认单** → 纯白话零术语（模板 `templates/confirmation-template.md`）："你要的是这个……当……时系统会……这次**不包含**……"。【开发拟定】的默认规则单独列出请业务重点过目。业务签认的是这份，它是日后"我没说过"时的依据。

最后：更新 context.md，提醒用户发确认单。

## 需求载体是现有系统/表格时（逆向场景）

"把老 Excel/老系统搬进新系统"极常见。此时**公式和代码即规格，但可能藏着已废弃的规则**（尤其按期发版的表格）。方法：

1. **先导出再逆向**：把公式导出为带单元格坐标的文本存 `raw/<文件名>-formulas.txt`（Excel 是二进制，导出件才可被证据核验引用）；再逐条翻译成"人话规则 + 判定条件 + 数据来源"，每条规则引用导出件行号；同构公式先归一化去重（上千行公式的独立规则往往只有几十条）。
2. **找规则 owner 验真**：规则生杀大权常不在提需求的人手里（成本模型归财务、审批链归管理层）。规则文档写入 `docs/requirements/rules/<来源名>.md`，做成 owner 能快速勾选的格式——每条留"有效 / 已废弃 / 需修改"勾选位；未验真的条目一律按【假设】对待。
3. **主动排查公式之外的隐性规则**：手工覆盖的单元格、隐藏 sheet/列、"某人每月手动调的那几行"——这类规则逆向不出来、owner 不被问也想不起来，是对数对不上的主要来源。列成"公式外可能存在的规则"清单，随规则文档一并验真。
4. **验收用"对数回归"**：足量历史数据（覆盖各分支、含边界案例）同输入下新老结果完全一致才通过；差异逐条由 owner 裁决是"老表旧错"还是"新系统新错"。
5. **警惕"逻辑应该都一致"类口头简化**——按【假设】处理，以逐条验真为准。

## 模式 B：需求变更

变更轮次同样适用证据纪律：变更原话先落盘 `raw/`，变更确认单与 spec 修改引用之，**每轮交付前重跑核验**（不只首次产出）。

**变更源判定（先做这步再动 spec）：** 同一来源的多次变更按时序覆盖——后者胜，前者废弃，不制造"冲突待决"；**只有明确来自不同人的意见才是冲突**，才生成二选一确认题。"又改了""还是按原来的"是强同源信号。不确定是否同源时，只问一个问题（"这两次是同一个人拍的板吗/谁说了算"），拿到答案前按同源暂处理，**绝不让 spec 因此挂起阻塞开发**。

1. **定位**：读相关 spec，找到变更触及的条目。
2. **影响分析**：受影响条目、哪些已确认答案失效需重确认、（结合代码）返工量。
3. **对变更本身再挑漏洞**：变更往往也是模糊的，同样选择题化。
4. **确认后**：更新 spec（旧决定标"已废弃，被 YYYY-MM-DD 变更取代"，不删除）；changes.md 记账；生成**写明返工代价**的变更确认单，让业务知情拍板。

## 模式 C：上下文维护

用户随口丢来的业务知识（"'单子'其实指采购单"）直接更新 context.md，简要回显。最轻的模式，别搞仪式感。

## 模式 D：链路审计（仅限有代码库时，主动找问题）

不需要用户先报障。方法必读 `references/chain-audit-checklist.md`：三份盘点（实体状态/操作入口/**不变量，钱优先**）→ 组合矩阵（按优先级收敛，一次审一族）→ 七种缺陷模式逐条对照 → 发现分流：守卫加固类直接给修复项；**规则未定义类选择题化进业务问题清单，走模式 A 阶段三管道**。审计沉淀进 specs/ 与 context.md，越审越快。

## 证据纪律（harness）

让"有据可查"从口头承诺变成可机检的声明。**证据源不限于代码**——仓库代码、raw/ 里归档的 PRD/聊天记录/回执/公式导出件都是证据源，同一脚本通吃。适用于一切引用来源的产出（阶段一结构化、阶段三确认记录、模式 D 审计、逆向规则文档）：

- **自证引用格式**：`> 证据: <路径>:<行号> | "<原文片段>"`。**路径一律以项目根为基准**（如 `docs/requirements/raw/xxx.md`，不是 `raw/xxx.md`），核验命令也从项目根跑 `--root .`——基准不一致会导致同一份产出换个环境判就翻车。片段是引用的自证材料——**没有片段的引用视同【假设】**。harness 只保证"引用保真"（来源没被篡改/编造），不保证"源头为真"（业务说的话可能是错的）——后者靠三档标签、落款与 owner 验真。
- **产出前必跑** `python3 scripts/verify_evidence.py <产出文件> --root <项目根>`（脚本同时机判：引用真伪、**零引用告警**、**无主数值探测**、逐维度覆盖行；正式交付可加 `--strict` 让无主数值直接判 FAIL）：FAIL 的引用要么修正、要么把对应结论降级为【假设】；**核验摘要必须附在产出末尾**，让用户一眼看到几条证据过检、有哪些假设在场。没跑核验的产出等于没做完。
- **阻塞级岔口必须配跨分支数值示例**：同一组具体输入，每个候选选项各算一遍、摆成对照表（"选 A 则第4期 4/1–4/30 不动；选 C 则撑到 5/31、后续全体后移"），并标注该示例**能区分/不能区分哪些选项**——区分不了就换例子。只算一个选项等于暗中替业务拍板；只画岔口不算数等于让拍板的人凭抽象想象做决定。示例中的数字一律属开发演示推算，需在表头注明"演示数字，非任何选项的背书"。
- **无主数值自检**：交付前逐一扫描产出中的具体数值、时间点、阈值、枚举——每一处要么有证据引用、要么有【】标签，二者皆无即为冒充。**没有证据引用支撑时，禁止使用"业务明示/业务已确认"类措辞**。此自检在时间压力下的降级模式里尤其不可跳过——无主数值是冒充最常见的形态。
- **覆盖强制（硬格式）**：阶段二和模式 D 的产出必须含逐维度结论行，每个维度独立一行、固定格式：`- [维度N 名称] 适用：发现X条问题` 或 `- [维度N 名称] 不适用：<一句话理由>`。**用总结段落替代逐行结论不算完成覆盖**——verify_evidence.py 会统计逐维度结论行并对不足 8 行告警。
